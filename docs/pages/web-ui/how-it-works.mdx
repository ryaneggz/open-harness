# How It Works

The web UI is a fully client-side application. There is no backend server in Phase 1 — all state lives in the browser and all LLM calls go directly from the browser to the provider API.

## Component architecture

The UI is built with [Lit](https://lit.dev/) web components via the `@mariozechner/mini-lit` wrapper. Components are standard custom elements that run in any modern browser without a framework runtime.

```
packages/web-ui/src/          # @openharness/web-ui component library
├── components/               # Reusable UI primitives (chat bubble, toolbar, etc.)
├── chat.ts                   # Main chat component
├── storage.ts                # IndexedDB adapter
└── index.ts                  # Public exports

packages/web-ui/example/src/  # pi-web-ui-example SPA
├── main.ts                   # Entry point — mounts <pi-chat> into #app
└── ...
```

The example app imports `@openharness/web-ui` (resolved as a workspace dependency during build) and mounts the chat component into a bare `index.html`. Vite bundles everything into `packages/web-ui/example/dist/`.

## Request flow

```
Browser
  └─► Settings dialog → reads API key from IndexedDB
  └─► <pi-chat> component → calls provider API directly (fetch)
        └─► Anthropic / OpenAI / Ollama / custom OpenAI-compatible endpoint
```

If the provider API does not send CORS headers, the component falls back to a CORS proxy automatically. No traffic passes through the nginx container — nginx only serves the static HTML/JS/CSS files.

## Storage model

All persistent state is kept in the browser's IndexedDB under the origin of the page:

| Store | Contents |
|-------|----------|
| `sessions` | Conversation history (messages, tool calls, results) |
| `settings` | Provider selection, model name, UI preferences |
| `api-keys` | Encrypted API keys per provider |

Clearing browser data (IndexedDB) permanently deletes all sessions and keys. There is no server-side backup in Phase 1.

## Build pipeline

The nginx image is built in two stages:

1. **builder** — `node:22-alpine` + pnpm installs workspace deps, runs `vite build` to produce `dist/`
2. **nginx:alpine** — copies `dist/` to `/usr/share/nginx/html`, writes a `default.conf` with `try_files $uri /index.html` for SPA routing

The build context is the repo root (one level above `.devcontainer/`), so pnpm can resolve workspace packages (`@openharness/web-ui`) during the install step.

## Routing

nginx is configured with a single `try_files` rule:

```nginx
location / {
    try_files $uri /index.html;
}
```

All unmatched paths fall back to `index.html`, which lets the Lit components handle client-side navigation.
